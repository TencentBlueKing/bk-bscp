# bscp 进程表达式过滤方案设计

> 关联需求：1020451610135740005（支持通配符（fnmatch）扩展的表达式语法过滤进程，短 ID 135740005）
> 前置梳理：`docs/reqs/gsekit进程过滤逻辑梳理.md`（gsekit 过滤逻辑权威梳理）
> 需求文档：`docs/reqs/进程表达式过滤.md`

## 1. 目标与约束

### 1.1 目标

让 bscp 支持与 gsekit **等价**的表达式过滤能力，保证同一条表达式在 bscp 与 gsekit 上命中
**相同的进程集合**（以 CC 进程 ID 标识），避免存量插件用户的表达式因语义偏差触发到非预期进程。

### 1.2 需求阶段已锁定的决策

| 项 | 决策 |
|----|------|
| 方案方向 | 只锁定"等价筛选效果 + 等价性回归测试"目标，A/B 实现路线由本设计确定 |
| 语法范围 | 全量对齐 gsekit：通配符 `*`/`?`、枚举 `[w1, w2]`、数字范围、单字符字母范围、排除 `[!seq]`、切片 `[a:b]`、组合 |
| 入口范围 | 插件路径 + 页面路径双入口支持表达式 |
| 进程 ID 语义 | 表达式进程 ID 段对齐 CC 进程 ID（`cc_process_id`） |

### 1.3 一致性基准

匹配语义以 gsekit 源码为权威基准：`bk-process-config-manager/apps/gsekit/utils/expression_utils/`
（`parse.py`/`match.py`/`range2re.py`/`serializers.py`）与 `.../process/handlers/process.py`。

## 2. 关键设计决策

### 2.1 决策一（回填 Q-001）：不新增 expression 字段，采用「内存拼接匹配」

**结论：采用方案 B'——查询时在内存把候选进程的 5 个字段拼成 expression 串，用等价的 Go
匹配内核匹配，得到命中的进程集合。不新增 DB 列、不改数据模型。**

理由：
1. gsekit 的 `expression` 列只是 5 个字段（集群名/模块名/服务实例名/进程别名/CC 进程 ID）
   的冗余拼接缓存，bscp 的 `Process` 表已逐列存储这些字段（`ProcessSpec` +
   `ProcessAttachment.CcProcessID`，见 `pkg/dal/table/process.go`）。
2. gsekit 本身就是**每次查询在内存**用 `fnmatch` 匹配全量 `biz + env` 进程
   （`ProcessHandler.expression_scope_to_scope`，`.../process.py`）。bscp 照此在内存匹配，
   语义与 gsekit 天然对齐，无需持久化。
3. 用「内存拼接同一条 expression 串 + 同一套 `gen_expression + list_match` 逻辑」可做到与
   gsekit **联合匹配语义逐字节等价**（含分隔符锚定、`*` 跨段等边界），同时零 schema 变更、
   零同步维护成本。

与方案 A（物化 expression 列）对比：

| 维度 | 方案 A（物化列） | 方案 B'（内存拼接，本方案选定） |
|------|-----------------|------------------------------|
| schema 变更 | 需加列 + migration | 无 |
| 同步维护 | 每次 CC 同步/字段变更需重算 expression | 无（查询时即时拼接） |
| 与 gsekit 语义一致性 | 高 | 高（同样高） |
| 重复查询性能 | 略优（省拼接） | 每次拼接（规模可控，见 §9） |
| 复杂度/风险 | 引入denormalize一致性风险 | 低 |

仅当 §9 性能实测在现网最大规模下成为瓶颈时，才回头考虑方案 A 作为优化（不在本期）。

### 2.2 决策二：fnmatch 保真——移植 CPython `fnmatch.translate` 到 Go regexp

**结论：新引擎的第 2 层匹配不复用 `github.com/gobwas/glob`、也不用 `path/filepath.Match`，
而是移植 CPython `fnmatch.translate` 的语义，将 fnmatch 模式翻译为 Go `regexp` 后匹配。**

理由（这是"筛选效果一致"的核心风险点）：
- gsekit 第 2 层用的是 Python 标准库 `fnmatch`，其否定字符集语法是 `[!seq]`；而
  `gobwas/glob`、`filepath.Match` 的语法/语义并不完全等价（否定写法、`*` 是否跨 `/`、
  字符集边界处理均有差异）。
- gsekit 第 1 层 `range2re` 会产出形如 `[1-9]`、`[0-9]` 的字符集片段，第 1 层排除会产出
  `[!ab]`，这些必须被第 2 层按 fnmatch 语义解释。只有忠实移植 `fnmatch.translate`
  （`*`→`.*`、`?`→`.`、`[!seq]`→`[^seq]`、`[seq]` 原样、整体 `(?s:...)\Z`）才能保证一致。
- 运行环境为 Linux，Python `fnmatch.filter/fnmatch` 经 `os.path.normcase` 为**大小写敏感**
  恒等，故 Go 侧用大小写敏感正则即可对齐。

## 3. 总体架构

```
                     ┌─────────────────────────────────────────┐
                     │  新增：表达式引擎  internal/expression    │
                     │  parse.go / match.go / range2re.go /      │
                     │  slice.go / scope.go / errors.go          │
                     │  （对齐 gsekit expression_utils）          │
                     └───────────────▲──────────────▲────────────┘
                                     │              │
        插件路径（等价 gsekit）        │              │   页面路径（能力增强）
  OperateProcess / 配置下发           │              │   ListProcess
        │                            │              │        │
        ▼                            │              │        ▼
  OperateRange(+表达式段)  ──► ExpressionScope 解析 ─┘   ProcessSearchCondition(+表达式)
        │                                                     │
        ▼                                                     ▼
  dao.GetByOperateRange ──► 内存候选(biz+env) ──► 引擎匹配 ──► 命中 cc_process_id 集合 ──► 归一
```

核心思想：**把"表达式 → 进程集合"的归一逻辑收敛到 `internal/expression` 引擎**，两条链路
都先取候选进程（按 `biz_id + environment` 及可下推的等值维度），再交引擎做表达式匹配，最终
落到进程 ID 集合，复用现有 DB 主流程。

## 4. 表达式引擎模块设计（`internal/expression`）

严格对齐 gsekit，函数一一对应，便于逐条比对与回归。

| Go 文件/函数（拟） | 对应 gsekit 源 | 职责 |
|-------------------|---------------|------|
| `range2re.go: Range2Re(begin, end int) []string` | `range2re.py: range2re` | 数字范围按位切割为正则字符集片段 |
| `parse.go: ParseExp2UnixShellStyle(expr string) ([]string, error)` | `parse.py: parse_exp2unix_shell_style` | 第 1 层 `[...]` 预处理，展开为 fnmatch 候选串列表 |
| `parse.go: getMatchType / parseEnumExpression / ...` | `parse.py` 同名 | 枚举/范围/排除/字面量类型判定与展开 |
| `match.go: Match(name, expr) (bool, error)` | `match.py: match` | 单值匹配（两层） |
| `match.go: ListMatch(names []string, expr) ([]string, error)` | `match.py: list_match` | 列表匹配，保持相对顺序 |
| `match.go: fnmatchTranslate(pat string) string` | CPython `fnmatch.translate` | fnmatch → Go regexp |
| `slice.go: SlicePattern / ExecuteSlice(names, sliceExpr) ([]string, error)` | `match.py: SLICE_PATTERN/execute_slice` | 切片 `[a:b]` 提取与结果切片 |
| `scope.go: ExpressionScope 结构 / GenExpression / ExpressionScopeToIDs` | `serializers.py: gen_expression` + `process.py: expression_scope_to_scope` | 五段拼接、缺省 `*`、切片单独处理、空集语义 |
| `errors.go` | `exceptions.py` | `ErrExpressionSyntax/Parse/Slice` 等价错误 |

关键数据结构：

```go
// ExpressionScope 对齐 gsekit ExpresssionScopeSerializer 的五段 + 环境
type ExpressionScope struct {
    Environment       string // bk_set_env，必填
    SetName           string // 缺省 "*"
    ModuleName        string // 缺省 "*"
    ServiceName       string // 缺省 "*"
    ProcessAlias      string // 缺省 "*"
    ProcessID         string // CC 进程 ID 段，支持切片，缺省 "*"
}
```

分隔符常量对齐 gsekit `EXPRESSION_SPLITTER`（`<-GSEKIT->`），仅用于内存拼接，不落库。

`ExpressionScopeToIDs` 主流程（对齐 `expression_scope_to_scope` 6 步）：
1. 取候选进程（`biz_id + environment`）的 5 字段，内存拼接 `expression` 串，建立
   `expression → cc_process_id` 映射。
2. 若进程 ID 段命中 `SlicePattern`，提取切片表达式并将该段临时置 `*`。
3. `GenExpression` 五段拼接为一条完整表达式。
4. `ListMatch(所有 expression, 完整表达式)` 返回命中子集。
5. 映射换回 CC 进程 ID 列表。
6. `ExecuteSlice` 对结果列表切片。

## 5. 数据流设计

### 5.1 插件路径（等价 gsekit，主目标）

- 入口：`OperateProcess`（进程操作）与配置下发 `buildConfigTemplateGroups`
  （`cmd/data-service/service/config_instance.go:219`）均经 `dao.GetByOperateRange`。
- 改造点：当请求携带表达式范围时，走引擎归一为 CC 进程 ID 集合，再查库；否则保持现有单值等值
  行为（兼容，见 §6）。
- 空集语义：表达式解析后命中为空 → 返回空集（与现有 `RecordNotFound` 处理衔接），**不**降级为
  全选。

### 5.2 页面路径（能力增强）

- 入口：`ListProcess` → `req.GetSearch()`（`ProcessSearchCondition`）→ `dao.List` →
  `handleSearch`（`internal/dal/dao/process.go:429`）。
- 改造点：`ProcessSearchCondition` 各维度支持表达式写法时，先经引擎把候选进程归一为命中 ID
  集合，再以 `id IN (...)` / `cc_process_id IN (...)` 参与现有分页查询；未使用表达式的字段
  保持原 `IN` 等值，保证向后兼容与分页正确。
- 优先级：等值枚举与表达式并存时，取用优先级对齐 gsekit（`scope` 优先于 `expression_scope`）。

## 6. 协议与兼容（回填 Q-003）

原则：**新增表达式能力，不破坏存量单值/等值调用。**

- 插件路径：新增与 gsekit 对齐的表达式范围结构（五段字符串 + `environment`）。存量调用仍走
  原 `OperateRange` 单值字段路径；两者择一，服务端按是否携带表达式范围分流。
  > 兜底认知：即便复用现有字段，literal 值在 fnmatch 下等价精确匹配；但为规避进程名中偶发的
  > `*?[]` 被解释，采用"新增表达式范围结构、与旧字段分流"的显式方案，避免语义漂移。
- 页面路径：`ProcessSearchCondition` 以增量方式支持表达式（不改变既有字段的等值语义）。
- proto 变更需重新生成桩代码并检查 diff（遵循仓库生成命令）。

> 具体字段命名与 proto 形态在实现前的接口评审确认；本设计锁定"分流、不破坏存量"的兼容策略。

## 7. 数据模型

**无变更**（决策 §2.1）。表达式匹配所需 5 字段已存在于 `Process` 表。

## 8. 匹配语义对齐要点（等价性 checklist）

实现与测试都以此为准（每条须有对应用例）：
- 两层解析：第 1 层 `[...]` 预处理产物可含 fnmatch 字符集，第 2 层 fnmatch 兜底。
- `[ab]` 字面量陷阱：无逗号/范围/`!` 的 `[ab]` 按字面量 `ab`，非字符集。
- 数字范围：`begin/end` 均十进制且 `int(begin) < int(end)`，`Range2Re` 按位切割逐位生成。
- 单字符字母范围：同大小写、单字符、`ord(begin) < ord(end)`；不满足退化为字面量。
- 排除 `[!seq]`：保留方括号，翻译为 `[^seq]`。
- 组合：`4[6, 8, 9]` → `46`/`48`/`49`（前缀 × 枚举笛卡尔积）。
- 切片 `[a:b]`：仅进程 ID 段，匹配前提取、匹配后对结果列表切片；`[-5:]` 等负索引对齐 Python
  切片语义。
- 缺省 `*`、`bk_set_env` 必填。
- 优先级（scope > expression）、空集语义（表达式空 → 空集，非全选）。
- 异常：非法表达式抛等价错误，不静默降级。
- 大小写敏感（Linux 恒等 normcase）。
- 顺序：`ListMatch` 结果保持相对候选列表顺序。

## 9. 性能与规模（回填 Q-002）

- 复杂度：单次过滤 = 载入 `biz+env` 候选进程 O(N) + 每候选串一次正则匹配 O(N×M)（M=第 1 层
  展开候选串数）。与 gsekit 同量级。
- 优化点：`fnmatchTranslate` 结果按模式缓存编译后的 `*regexp.Regexp`；第 1 层展开去重。
- 待实测确认（Q-002）：单业务进程最大规模量级、单次过滤 P95 目标；据此判断是否需要方案 A 或
  下推部分等值维度到 DB 以缩小候选集。

## 10. 测试策略（对齐需求 F-005 / AC-002）

- **单元测试**：对引擎每个函数按 §8 checklist 编写用例，可直接复用 gsekit
  `expression_utils/tests.py` 的语料作为黄金用例。
- **等价性回归测试**：构造进程数据集 + 表达式语料集，断言 bscp 引擎命中的 CC 进程 ID 集合
  （含切片顺序）与 gsekit 一致。语料覆盖 §8 全部条目。
- 单包测试优先（`go test ./internal/expression/...`），不只依赖全量编译。

## 11. 实现计划（任务拆分）

1. `internal/expression` 引擎：`range2re.go` + 单测（先行，独立可验证）。
2. `parse.go`（第 1 层）+ `match.go`（`fnmatchTranslate` + `Match`/`ListMatch`）+ 单测。
3. `slice.go` + `scope.go`（`ExpressionScope`/`GenExpression`/`ExpressionScopeToIDs`）+ 单测。
4. 等价性回归测试（移植 gsekit 语料）。
5. proto 与桩代码：新增表达式范围结构，重新生成并检查 diff。
6. 插件路径接入：`GetByOperateRange` 分流表达式范围 → 引擎归一。
7. 页面路径接入：`ListProcess`/`handleSearch` 增量支持表达式。
8. 联调 + 性能实测（回填 Q-002）。

每步保持小改动、可单独测试；引擎（1–4）与接入（5–8）解耦，引擎先独立达成等价再接线。

## 13. 风险

- fnmatch 语义偏差（最高风险）：靠 §2.2 忠实移植 + §10 等价性回归兜底。
- `range2re` 边界（进位/交集切割）移植错误：以 gsekit 语料黄金用例逐条比对。
- 页面路径表达式与分页/其他等值维度叠加的语义组合：以 §5.2 优先级与 checklist 覆盖。
- 大候选集性能：见 §9，必要时下推等值维度或转方案 A。
