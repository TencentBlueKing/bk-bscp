## 需求背景

### 业务背景

随着 BSCP 系统引入**项目（Project）**和**环境（Environment）**维度，现有的密钥规则（credential_scopes）功能需要进行相应的增强，以支持新的多级作用域模型：

- **当前痛点**：密钥规则（credential_scopes）表缺少项目和环境维度字段，无法精确控制不同项目/环境下的密钥访问权限
- **目标**：为 credential_scopes 表添加 project_id 和 environment_id 字段，并在 API 层面支持这些维度的查询和返回
- **业务价值**：实现更细粒度的密钥权限管理，支持在不同项目和环境下灵活配置密钥访问规则

### 详细功能描述

#### [F-001] UpdateCredentialScopes 增强项目和环境字段

- **输入**：
  - `UpdateCredentialScopesReq` 新增 `project_id` 字段（第435行）
  - `CredentialScopeSpec` 新增 `env_id` 字段（proto 第27行）
  - `UpdateScopeSpec` 新增 `env_id` 字段（proto 第36行）
- **处理逻辑**：
  1. 接收请求参数，包含 biz_id、credential_id、project_id
  2. 遍历 created 列表，为每个 CredentialScope 设置 Attachment.ProjectID 和 Attachment.EnvID
  3. 遍历 updated 列表，为每个 CredentialScope 更新 Attachment.ProjectID 和 Attachment.EnvID
  4. 在事务中执行创建、更新、删除操作
  5. 更新 credential 的 revision 信息
- **输出**：`UpdateCredentialScopesResp` (空响应)
- **边界条件**：
  - project_id 可以为空（兼容旧数据）
  - env_id 可以为空（兼容旧数据）
- **异常处理**：
  - credential 不存在 → 返回错误
  - 事务失败 → Rollback 并返回错误

**代码位置**：`cmd/data-service/service/credential_scope.go:80-181`

#### [F-002] ListCredentialScopes 返回环境信息

- **输入**：
  - `ListCredentialScopesReq` 包含 biz_id、credential_id、project_id（第467-470行）
- **处理逻辑**：
  1. 验证 credential 是否存在
  2. 查询 credential_scopes 表获取所有关联的 scope 记录
  3. 收集所有非空的 env_id
  4. 批量查询 environments 表获取环境详细信息（ID、Name、Type）
  5. 构建 envMap[uint32]*table.EnvironmentSpec
  6. 调用 PbCredentialScopes 将环境信息填充到 CredentialScopeSpec 中
- **输出**：`ListCredentialScopesResp` 包含 count 和 details 列表
  - 每个 CredentialScopeList 包含：
    - id: scope ID
    - spec: 包含 app, scope, **env_id**, **env_name**, **env_type**
    - attachment: 包含 biz_id, credential_id, **project_id**, **env_id**
    - revision: 审计信息
- **边界条件**：
  - env_id 为 0 或空时不查询环境信息
  - 环境被删除时 envMap 中无对应记录，显示空值
- **异常处理**：
  - credential 不存在 → 返回错误
  - 环境查询失败 → 返回错误

**代码位置**：`cmd/data-service/service/credential_scope.go:31-76`

## 业务规则

### 业务逻辑规则

- **规则 R-001**：新增密钥规则时，必须同时保存 project_id 和 env_id（如果调用方提供）

  - project_id: uint32 (> 0 表示有效值，0 或空表示未设置)
  - env_id: uint32 (> 0 表示有效值，0 或空表示未设置)

### 数据库迁移策略

**迁移版本**：20260529100000_add_project_env_dimension

**执行步骤**（共5步）：
1. **Step1**: 创建 projects / environments / scope_migration_tasks 表 + 注册 id_generators
2. **Step2**: 为每个 biz 创建默认项目(key=default) 和默认环境(type=prod)
3. **Step3**: 为 credential_scopes 等 EnvScope 表添加 project_id + environment_id 字段（AutoMigrate）
4. **Step4**: 调整已有唯一索引（加入 project_id/environment_id）
5. **Step5**: 分批回填存量数据的 project_id/environment_id（幂等、可续跑）

**回填机制特点**：
- 使用 scope_migration_tasks 表记录进度（last_id + status）
- 按 ID 分批（batchSize=1000），单批失败不影响整体进度
- 支持断点续跑（从 last_id 继续）
- 多实例安全（INSERT IGNORE + status 字段防并发）
