## 需求背景

### 业务背景

bscp 引入"项目与环境（project / environment）"维度后，前端 UI（`ui/`）随之改版，新增了项目管理、环境管理、业务/项目选择器等页面与交互。

问题在于：**存量用户仍在旧环境的使用流程中，其心智与操作路径基于旧版 UI**。若直接把所有用户切到新 UI，会打断存量用户的既有使用习惯、带来迁移风险。

因此需要在服务侧**同时部署两套 UI**：

- 存量用户继续访问**旧 UI**（不含项目/环境概念，代码位于 `ui_old/`）；
- 新用户 / 新环境访问**新 UI**（含项目/环境概念，代码位于 `ui/`）。

由于当前前端通过 `go:embed ui/dist` 在**编译期**嵌入 `bk-bscp-ui` 二进制，"两套 UI 并存"必然涉及构建与部署链路的调整（Makefile、embed、Chart 等）。本需求聚焦在**测试环境**先跑通两套 UI 的构建与部署，并完成相应的构建/发布链路改造，为后续生产上线打基础。

### 用户故事

作为**存量业务用户**
我想要在改版后仍通过原有入口访问旧版 UI
以便于不打断我既有的配置管理使用习惯

作为**新接入/新环境用户**
我想要访问含"项目与环境"能力的新版 UI
以便于使用项目、环境维度的配置管理能力

作为**平台运维/发布人员**
我想要通过统一的构建与 Chart 部署能力同时发布新旧两套 UI
以便于在测试环境验证双 UI 并存方案、并为生产上线做准备

### 需求来源

- **需求渠道**：产品规划（项目与环境管理专项）

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 旧 UI（`ui_old/`）接入前端构建链，产出可部署的静态资源 | P0 | 运维/研发 | 必须 |
| F-002 | 构建产物能同时承载新旧两套前端（编译期嵌入） | P0 | 研发 | 必须 |
| F-003 | UI 服务支持按"前端变体（新/旧）"启动，渲染对应前端 | P0 | 研发 | 必须 |
| F-004 | Helm Chart 支持部署两套 UI（新/旧各一套 Deployment/Service/ConfigMap） | P0 | 运维 | 必须 |
| F-005 | 测试环境按**访问域名**将流量分流到新 UI 或旧 UI 服务 | P0 | 运维 | 必须 |
| F-006 | 两套 UI 共用同一后端 apiserver 及数据 | P0 | 研发/运维 | 必须 |

### 详细功能描述

#### [F-001] 旧 UI 接入构建链

- **输入**：`ui_old/` 源码
- **处理逻辑**：
  1. 为 `ui_old/` 增加前端构建入口（npm 构建）
  2. 产出旧 UI 静态资源目录（如 `ui_old/dist`）
- **输出**：可被镜像嵌入/托管的旧 UI 静态资源
- **边界条件**：旧 UI 源码保持现状，不做功能改造，仅接入构建
- **异常处理**：旧 UI 构建失败时，构建流程需明确报错、不产出污染产物

#### [F-003] UI 服务按变体渲染

- **输入**：UI 服务启动参数（前端变体：新 / 旧）+ 运行时配置（route_prefix、API 地址等）
- **处理逻辑**：
  1. 服务启动时根据变体参数确定使用哪套前端静态资源
  2. 沿用现有 `index.html` 运行时注入（`BK_STATIC_URL` / `API_URL` / `SITE_URL` 等）
- **输出**：对应变体的 UI 页面
- **边界条件**：两套 UI 共用同一后端接口地址
- **异常处理**：变体参数缺失/非法时，服务启动应失败并给出明确提示

#### [F-005] 域名分流

- **输入**：用户访问的域名
- **处理逻辑**：测试环境入口（Ingress/网关）按域名将请求转发到新 UI Service 或旧 UI Service
- **输出**：用户根据访问域名落到对应 UI
- **边界条件**：两套 UI 共享同一后端；域名与各自 `route_prefix` 需对齐
- **异常处理**：域名未匹配到任一 UI 时返回明确的路由错误

## 非功能需求

### 性能需求

- 双 UI 并存不改变单个 UI 的既有性能表现（本需求不引入额外性能指标要求）。

### 安全需求

- **权限控制**：沿用现有 UI 鉴权（`webAuthentication`），双 UI 均需鉴权，不新增越权面。
- **数据保护**：两套 UI 共用同一后端与数据，不引入新的数据出口。

### 可用性与稳定性

- 新旧 UI 为**独立 Deployment/Service**，其中一套异常不影响另一套。

### 兼容性

- **接口兼容**：两套 UI 调用同一 apiserver，后端接口需对新旧 UI 均兼容。
- **部署兼容**：Chart 改造需兼容现有单 UI 部署方式（不破坏存量部署）。

## 业务规则

### 业务逻辑规则

- **规则 R-001**：存量用户访问旧 UI，新用户/新环境访问新 UI，路由维度为**访问域名**。
- **规则 R-002**：两套 UI 仅前端不同，后端 apiserver 与数据共用同一套。
- **规则 R-003**：旧 UI 仅作并存过渡，本需求不对旧 UI 做功能改造。

### 权限规则

- 新旧 UI 均沿用现有登录与鉴权体系，不新增角色或权限模型。

### 部署验收

- [ ] **AC-D01**：新旧两套 UI 为独立 Deployment，互不影响；单套重启不影响另一套访问

## 边界范围

### 本期包含

- 旧 UI（`ui_old/`）接入前端构建链
- 构建产物同时承载新旧两套前端（编译期嵌入）
- UI 服务支持按前端变体启动
- Helm Chart 支持双 UI 部署（新/旧各一套 Deployment/Service/ConfigMap）
- 在**测试环境**跑通双 UI 部署，并按域名完成流量分流
- 完成 Makefile / embed / Chart 相关构建发布链路改造

### 本期不包含

- **生产环境部署**（另立需求）
- 旧 UI 的功能改造
- 存量用户→新 UI 的迁移策略与旧 UI 下线计划（后续需求）
- 新旧 UI 后端/数据隔离（本期共用同一后端）

## 约束条件

- **技术方向（初步，设计阶段细化）**：采用单一 hyper 镜像同时嵌入新旧两套前端静态资源，通过启动参数选择前端变体；测试环境以两套 Deployment/Service + 按域名路由承载，共用同一后端 apiserver。
- **技术限制**：前端经 `go:embed` 于编译期嵌入，静态资源变化需重新构建镜像。
- **兼容限制**：Chart 改造不得破坏现有单 UI 部署。
