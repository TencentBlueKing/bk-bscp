## 需求背景

在同一镜像可按变体启动新/旧 UI 的基础上（前置子需求），需扩展 Helm Chart（`bcs-bscp/`）以在测试环境部署两套 UI 服务，并按访问域名将流量分流到对应 UI；两套 UI 共用同一后端 apiserver 与数据。同时改造不得破坏现有单 UI 部署方式。

## 用户故事

作为**平台运维/发布人员**
我想要通过 Helm Chart 一次部署新旧两套 UI 并按域名分流
以便于存量用户访问旧 UI、新用户访问新 UI，且不破坏现有部署

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 |
|---------|---------|--------|
| F-004 | Chart 支持部署两套 UI（新/旧各一套 Deployment/Service/ConfigMap） | P0 |
| F-005 | 测试环境按访问域名将流量分流到新 UI 或旧 UI 服务 | P0 |
| F-006 | 两套 UI 共用同一后端 apiserver 及数据 | P0 |

### 详细功能描述

#### [F-004] Chart 双 UI 部署

- **输入**：Helm values（镜像、各 UI 变体、`route_prefix` 等）
- **处理逻辑**：
  1. 扩展 Chart，新增旧 UI 的 Deployment/Service/ConfigMap（或参数化复用现有模板）
  2. 两套 Deployment 使用同一镜像、不同启动变体参数与 `route_prefix`
  3. values 增加旧 UI 配置块，保持默认不启用以兼容存量单 UI 部署
- **输出**：集群内新旧两套 UI 的 Deployment/Service
- **边界条件**：默认关闭旧 UI 时，部署行为与现状一致
- **异常处理**：配置缺失/冲突时 Helm 渲染报错并提示

#### [F-005] 域名分流

- **输入**：用户访问的域名
- **处理逻辑**：测试环境入口（Ingress/网关）按域名转发到新 UI Service 或旧 UI Service
- **输出**：用户按访问域名落到对应 UI
- **边界条件**：域名与各自 `route_prefix` 需对齐
- **异常处理**：域名未匹配到任一 UI 时返回明确路由错误

#### [F-006] 共用后端

- **处理逻辑**：两套 UI 的 `API_URL` 指向同一 apiserver
- **边界条件**：本期不做后端/数据隔离
- **异常处理**：后端不可用时按现有 UI 错误提示处理

## 验收标准

- [ ] **AC-001**：Given 使用改造后的 Chart 部署到测试环境 When 部署完成 Then 集群中存在新旧两套 UI 的 Deployment/Service 且均 Ready
- [ ] **AC-002**：Given 已配置域名分流 When 用旧 UI 域名访问 Then 落到旧 UI；用新 UI 域名访问 Then 落到新 UI
- [ ] **AC-003**：Given 两套 UI 均已部署 When 在任一 UI 执行配置查询 Then 请求命中同一后端 apiserver 并正确返回
- [ ] **AC-004**：Given Chart 改造完成 When 以原单 UI 方式部署（旧 UI 关闭） Then 存量部署方式仍可用（向后兼容）
- [ ] **AC-D01**：Given 两套 UI 独立 Deployment When 单套重启 Then 不影响另一套访问

## 边界范围

### 本期包含

- Chart 新增旧 UI 的 Deployment/Service/ConfigMap 与 values 参数化
- 测试环境按域名分流
- 两套 UI 共用同一后端
- 向后兼容现有单 UI 部署

### 本期不包含

- 生产环境部署（另立需求）
- 新旧 UI 后端/数据隔离
- 存量用户迁移策略与旧 UI 下线计划
