## 需求背景

bscp 新 UI（`ui/`）改版引入项目/环境能力后，存量用户需继续使用旧 UI（`ui_old/`）。当前前端经 `go:embed ui/dist` 于编译期嵌入 `bk-bscp-ui` 二进制，仅嵌入了新 UI。本子需求解决**构建链层**问题：让旧 UI 可被构建，并使构建产物同时承载新旧两套前端静态资源，为后续服务按变体启动、双 UI 部署打基础。

## 用户故事

作为**平台运维/发布人员**
我想要一次构建即可产出同时含新旧两套前端的产物
以便于后续无需分别维护两条镜像构建线即可部署双 UI

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 |
|---------|---------|--------|
| F-001 | 旧 UI（`ui_old/`）接入前端构建链，产出静态资源目录（如 `ui_old/dist`） | P0 |
| F-002 | `embed.go` 同时嵌入 `ui/dist` 与 `ui_old/dist`；Makefile 增加旧 UI 构建 target | P0 |

### 详细功能描述

#### [F-001] 旧 UI 接入构建链

- **输入**：`ui_old/` 源码
- **处理逻辑**：
  1. 为 `ui_old/` 补齐/适配前端构建入口（npm 构建）
  2. 产出旧 UI 静态资源目录（如 `ui_old/dist`）
- **输出**：可被 `go:embed` 嵌入的旧 UI 静态资源
- **边界条件**：旧 UI 源码保持现状，不做功能改造，仅接入构建
- **异常处理**：旧 UI 构建失败时明确报错，不产出污染产物

#### [F-002] 双前端产物嵌入

- **输入**：`ui/dist`、`ui_old/dist`
- **处理逻辑**：
  1. `embed.go` 增加对 `ui_old/dist` 的嵌入
  2. Makefile 增加旧 UI 构建 target（如 `build_frontend_old`），并纳入整体构建流程
- **输出**：单一二进制内同时含新旧两套前端静态资源
- **边界条件**：不改变现有新 UI 构建行为；CI 占位文件机制需兼容新增嵌入目录
- **异常处理**：任一前端产物缺失时构建应失败并提示

## 验收标准

- [ ] **AC-001**：Given 旧 UI 已接入构建链 When 执行旧 UI 构建 Then 产出 `ui_old/dist` 且无报错
- [ ] **AC-002**：Given `embed.go` 已嵌入两套前端 When 编译 `bk-bscp-ui` Then 编译通过且二进制内同时含新旧两套静态资源
- [ ] **AC-003**：Given 现有新 UI 构建流程 When 执行原构建 Then 新 UI 构建行为不受影响（向后兼容）
