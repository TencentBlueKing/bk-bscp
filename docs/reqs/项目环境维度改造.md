## 需求背景

### 业务背景

BSCP 系统需要承接游戏业务场景，新增**项目（Project）**和**环境（Environment）**两个维度，实现多项目、多环境的资源隔离能力：

- **当前问题**：系统仅支持业务（Biz）单一维度，无法满足游戏业务多项目并行开发的需求
- **目标用户**：使用 BSCP 进行配置管理的游戏业务研发团队
- **核心价值**：
  - 项目维度：用于隔离**服务（App）、脚本、分组、客户端、密钥、模板**等资源
  - 环境 + 项目维度：用于管理 **KV 配置项和配置文件（ConfigItem）** 的多环境发布

### 用户故事

> 作为 BSCP 的使用者，
> 我想要在所有服务的 CRUD 操作中携带 `project_id` 和 `environment_id`，
> 以便系统能够正确地将资源归属到对应的项目和环境下，实现多租户的资源隔离。

### 详细功能描述

#### [F-001] API 路由层改造

- **输入**：HTTP/gRPC 请求
- **处理逻辑**：
  1. 新路由：请求参数中显式传入 `project_id` 和 `environment_id`
  2. 旧路由：不携带这两个字段，系统自动使用**默认项目**和**默认环境**
  3. 中间件层统一解析并注入到 Kit 上下文中
- **输出**：携带完整维度的 Kit 上下文传递给下游服务
- **边界条件**：
  - 请求同时携带了 `project_id` 但未携带 `environment_id` → 使用指定项目的默认环境
  - 两个字段都未携带 → 使用全局默认项目和默认环境
- **异常处理**：
  - `project_id` 不存在 → 返回错误码：项目不存在
  - `environment_id` 不存在或不属于该项目 → 返回错误码：环境不存在或不在该项目中

#### [F-002] ConfigServer 接口改造

涉及接口范围（**本期仅改造以下模块**）：

| 接口模块 | 操作类型 | 说明 |
|---------|---------|------|
| App（服务） | Create/List/Update/Delete | 服务管理 |
| ConfigItem（配置项） | Create/List/Update/Delete | 配置文件管理 |
| KV（键值对） | Create/List/Update/Delete/Set | KV配置管理 |
| Publish（发布） | Create/Confirm/Rollback | 发布管理 |

> **注意**：Strategy（发布策略）不在本期范围。

- **输入**：从 Kit 中获取 `project_id` 和 `environment_id`
- **处理逻辑**：
  1. 所有 Create 操作：将 `project_id` 和 `environment_id` 写入数据记录
  2. 所有 Query 操作：增加 `WHERE project_id = ? AND environment_id = ?` 条件
  3. 所有 Update/Delete 操作：先校验资源的 `project_id` 和 `environment_id` 是否匹配
- **输出**：操作结果 + 数据记录（含完整的维度信息）

#### [F-003] DataService 接口改造

涉及接口范围（**本期仅改造以下模块**）：

| 接口模块 | 操作类型 | 说明 |
|---------|---------|------|
| App（服务） | Get/List/Create/Update/Delete | 服务管理 |
| ConfigItem | Get/List/Create/Update/Delete | 配置文件管理 |
| KV | Get/List/Create/Update/Delete | KV配置管理 |
| Release（生成版本/发布版本） | Get/List/Create | 版本管理 |
| ReleasedCI（已发布配置项） | Get/List | 已发布内容查询 |
| ReleasedKV（已发布KV） | Get/List | 已发布KV查询 |

> **注意**：Credential、Strategy 不在本期范围。CacheService 改造不在本期范围。

- **输入/输出逻辑同 F-002**

- **新增/修改点**：
  1. 各实体的 List/Query 方法增加 `projectID` 和 `envID` 参数
  2. 各实体的 Create 方法自动填充 `projectID` 和 `envID`
  3. 各实体的 Update/Delete 方法增加维度校验

#### [F-005] 默认值回退机制

- **默认项目**：`GetDefaultProject(kit, bizID)` 返回 `is_default = true` 的项目
- **默认环境**：`GetDefaultEnvironment(kit, bizID, projectID)` 返回 `name = "default"` 的环境
- **触发场景**：当请求未携带 `project_id` 或 `environment_id` 时自动启用
- **实现位置**：API Server Middleware 或各 Service 层入口

## 数据模型

### 已有的核心实体

#### Project（项目表: projects）

```go
type Project struct {
    ID         uint32             `gorm:"primaryKey"`
    Spec       *ProjectSpec       `gorm:"embedded"`
    Attachment *ProjectAttachment `gorm:"embedded"`
    Revision   *Revision          `gorm:"embedded"`

type ProjectSpec struct {
    Name      string       // 名称
    Key       string       // 标识 (BK-BSCP-XXXXX)
    Memo      string       // 描述
    Protected bool         // 是否保护
    IsDefault sql.NullBool // 是否默认项目
}

type ProjectAttachment struct {
    TenantID string // 租户ID
    BizID    uint32 // 业务ID
}
```

#### Environment（环境表: environments）

```go
type Environment struct {
    ID         uint32                 `gorm:"primaryKey"`
    Spec       *EnvironmentSpec       `gorm:"embedded"`
    Attachment *EnvironmentAttachment `gorm:"embedded"`
    Revision   *Revision              `gorm:"embedded"`
}

type EnvironmentSpec struct {
    Name         string          // 环境名称
    Type         EnvironmentType // prod/staging/test/dev
    Memo         string          // 描述
    DisplayOrder uint32          // 排序
    Protected    bool            // 是否保护
}

type EnvironmentAttachment struct {
    TenantID  string // 租户ID
    BizID     uint32 // 业务ID
    ProjectID uint32 // 所属项目ID ← 外键关联 Project
}
```

#### App（应用表: applications）— 已改造

## 业务规则

### 业务逻辑规则

- **规则 R-001**：项目维度覆盖的资源类型：App（服务）、脚本、分组、客户端、密钥、模板
- **规则 R-002**：环境 + 项目维度覆盖的资源类型：KV 配置项、配置文件（ConfigItem）
- **规则 R-003**：每个业务（Biz）必须有且仅有一个默认项目（is_default = true）
- **规则 R-004**：每个项目下必须有且仅有一个默认环境（name = "default"）
- **规则 R-005**：旧路由调用时，自动降级使用默认项目和默认环境
- **规则 R-006**：新路由调用时，必须显式提供有效的 project_id 和 environment_id

### 权限规则

- **规则 R-007**：跨项目的资源不可见（List 接口自动按 project_id 过滤）
- **规则 R-008**：跨环境的资源在特定场景可见（如 Release 列表可查看所有环境的发布历史）
