## 需求背景

### 业务背景

bscp（蓝鲸基础配置平台）在「进程配置管理」场景下需要按条件过滤进程。当前 bscp 仅支持
**等值枚举匹配**：

- 页面路径：`ProcessSearchCondition` 各字段（集群、模块、服务实例、进程别名、cc 进程 ID、
  内网 IP 等）走 DB 的 `IN` 等值过滤。
- 插件路径：`OperateRange` 各字段（`set_name`/`module_name`/`service_name`/`process_alias`/
  `cc_process_id`）走 DB 的**单值等值**过滤。

gsekit（进程配置管理，bk-process-config-manager）在相同场景下，插件请求路径支持「表达式」
匹配：通配符、数字/字母范围、枚举列表、排除、切片等复杂匹配（详见前置梳理文档）。

存量插件用户当前已在 gsekit 上使用这套表达式来圈定进程。若 bscp 的过滤语义与 gsekit 存在
偏差，**同一条表达式在 bscp 与 gsekit 上可能命中不同的进程集合**，进而导致进程操作
（启动/停止/下发配置等）作用到非预期进程，产生生产风险。因此 bscp 需要支持与 gsekit 等价
的表达式过滤能力，保证筛选效果一致，降低存量插件用户的迁移与理解成本。

### 用户故事

作为使用进程配置管理插件的存量调用方
我想要用与 gsekit 完全相同的表达式在 bscp 上筛选进程
以便于同一条表达式在两侧命中相同的进程集合，避免操作触发到非预期进程。

作为在 bscp 页面进行进程配置管理的用户
我想要在页面过滤进程时也能使用表达式（通配符/枚举/范围等）
以便于用更灵活的条件快速圈定目标进程，而不局限于等值枚举勾选。

作为 bscp 的研发/测试人员
我想要一套以 gsekit 语料为准的等价性回归测试
以便于持续验证 bscp 表达式过滤与 gsekit 结果一致，防止语义回退。

### 详细功能描述

#### [F-001] 表达式解析与匹配内核

- **输入**：单个字段的表达式字符串（如 `proc*`、`[管控平台, PaaS平台]`、`4[6, 8, 9]`、
  `[1-1000]`、`[!ab]`、`[0:10]`）。
- **处理逻辑**（与 gsekit 两层解析等价，详见梳理文档第五章）：
  1. 第 1 层 `[...]` 预处理：从左到右扫描方括号块，块外文本作前缀，块内按类型展开
     （逗号词列表 / 数字范围 / 单字符字母范围 / 排除 `!` / 普通词字面量），多块做笛卡尔积，
     产出若干候选串；数字范围按位切割为字符集正则片段（等价 `range2re`），单字符字母范围与
     排除保留方括号形成 fnmatch 字符集。
  2. 第 2 层 `fnmatch` 兜底：每条候选串用 Unix shell 通配语义匹配（`*`/`?`/`[seq]`/`[!seq]`），
     任一命中即命中。
- **输出**：某字段值是否匹配该表达式（布尔）/ 命中的值子集（保持原相对顺序）。
- **边界条件**：
  - `[ab]` 字面量陷阱：无逗号/范围/`!` 的 `[ab]` 按字面量 `ab` 处理，不是字符集。
  - 数字范围需 `begin/end` 均十进制且 `int(begin) < int(end)`；单字符字母范围需同大小写、
    单字符、`ord(begin) < ord(end)`；不满足退化为普通词字面量。
- **异常处理**：
  - 表达式语法非法（如缺少 `]`、范围解析错误、切片解析错误）→ 抛出对齐 gsekit 的解析异常
    （`ExpressionSyntaxException`/`ExpressionParseException`/`ExpressionSliceException` 的等价语义），
    返回明确错误信息，不静默降级为全选或空集。

#### [F-002] 插件路径支持表达式

- **输入**：插件请求，携带 gsekit 风格的分字段表达式（集群名 / 模块名 / 服务实例名 /
  进程别名 / 进程 ID）+ 环境类型（`bk_set_env`，必填）。
- **处理逻辑**：
  1. 每字段缺省为 `*`（匹配任意）。
  2. 各字段表达式按 F-001 解析并匹配对应进程字段。
  3. 进程 ID 段按 F-004 处理（含切片）。
  4. 归一为目标进程集合，供后续进程操作使用。
- **输出**：命中的进程集合。
- **边界条件**：`bk_set_env` 必填；解析后命中为空 → 返回空集（不视为全选）。
- **异常处理**：表达式非法 → 返回明确错误，不误触发到全量进程。

#### [F-003] 页面路径支持表达式

- **输入**：页面进程列表过滤条件，各字段支持表达式写法。
- **处理逻辑**：与 F-002 同一套解析/匹配内核；页面同时保留原有等值枚举勾选能力，二者语义与
  gsekit 页面路径归一到进程集合的效果对齐。
- **输出**：过滤后的进程列表（分页）。
- **边界条件**：与 gsekit 一致的优先级与空值语义（见业务规则 R-002/R-003）。

#### [F-004] 进程 ID 段对齐 CC 进程 ID + 切片

- **输入**：进程 ID 段表达式，可含切片 `[a:b]`（如 `[0:10]`、`[-5:]`）。
- **处理逻辑**：进程 ID 段匹配对象为 `cc_process_id`（与 gsekit 的 `bk_process_id` 一致）；
  切片单独处理——匹配前提取切片、该段临时置 `*`，匹配得到进程 ID 列表后再对结果列表做切片。
- **输出**：切片后的进程 ID 列表。
- **边界条件**：切片语法 `[a:b]` 仅对进程 ID 段生效，走独立于两层解析的切片分支。

#### [F-005] 等价性回归测试

- **输入**：一组以 gsekit 为基准的表达式语料 + 进程数据集。
- **处理逻辑**：对每条语料，比对 bscp 与 gsekit 命中的进程集合（及顺序，如涉及切片）。
- **输出**：一致性测试结果（全部一致为通过）。
- **边界条件**：语料需覆盖通配符、枚举、数字/字母范围、排除、切片、组合写法、`[ab]` 字面量
  陷阱、空集与异常等关键场景。

## 非功能需求

### 性能需求

- **响应时间**：单次表达式过滤（含解析 + 匹配 + 归一为进程集合）在单业务进程规模下的
  P95 响应时间不劣于当前等值枚举过滤的量级。具体阈值待确认（Q-002）。
- **数据容量**：需支持单业务进程数量达到现网最大规模（具体数量级待确认，Q-002）。

### 安全需求

- **权限控制**：沿用 bscp 现有进程配置管理的鉴权与业务隔离（`biz_id` 维度），本需求不改变
  权限模型。
- **数据保护**：表达式输入需做合法性校验，避免因超大范围/超长表达式导致的资源放大或异常。

### 兼容性

- **接口兼容**：插件路径改造需保持对存量插件调用的向后兼容——既有等值/单值调用行为不被破坏，
  同时新增表达式匹配能力（具体兼容形态由技术设计确定，见 Q-001）。
- **语义兼容**：匹配语义以 gsekit 为基准对齐，保证同一表达式两侧命中一致。

## 业务规则

### 业务逻辑规则

- **规则 R-001（语法全量对齐）**：bscp 支持的表达式语法与 gsekit 全量对齐，包括通配符
  `*`/`?`、枚举 `[w1, w2]`、数字范围 `[a-b]`、单字符字母范围、排除 `[!seq]`、切片 `[a:b]`
  及前缀+枚举组合（如 `4[6, 8, 9]`）。
- **规则 R-002（优先级）**：等值枚举范围与表达式范围同时存在时的取用优先级，与 gsekit 对齐
  （gsekit 中 `scope` 优先于 `expression_scope`）。
- **规则 R-003（空值语义）**：与 gsekit 对齐——等值枚举中某字段为空=该维度不过滤（全选）；
  表达式解析后进程集合为空=无可选数据，返回空集（非全选）。
- **规则 R-004（进程标识）**：表达式的进程 ID 段以 CC 进程 ID（`cc_process_id`）为准。

### 数据校验规则

- **必填字段**：环境类型（`bk_set_env`/`environment`）在插件表达式路径必填。
- **格式要求**：各字段表达式为字符串，缺省 `*`；切片仅进程 ID 段可用。
- **取值范围**：数字范围/字母范围需满足 F-001 的合法性判定，否则退化为字面量或抛异常。

## 澄清记录

### 第 1 轮澄清

**提问时间**：2026-07-02 22:56

**用户提问**：请评估是否还需要延续 gsekit 的方案为每条数据添加一个 expression 字段，
以保证与 gsekit 一样的筛选效果，避免存量插件表达式触发不一样的进程。

**Agent 评估结论**：
1. **不需要为「筛选效果一致」而必须新增 expression 字段**。gsekit 的 `expression` 列只是 5 个
   字段的冗余拼接缓存，而这 5 个字段 bscp 已逐列存储；gsekit 过滤在内存用 `fnmatch` 完成，
   expression 列只是把 5 段拼成一条串以便一次 `list_match`。
2. gsekit 用稀有分隔符 `<-GSEKIT->` 拼接，本质是「用一次 fnmatch 做 5 段独立字段匹配」的技巧；
   只要字段值不含该分隔符（现实数据不含），「拼接整体匹配」与「逐段匹配对应字段」结果等价。
   故 bscp 有两条保真度相同的路线（方案 A 物化字段 / 方案 B 逐段匹配）。
3. **决定一致性的是匹配语义的逐条对齐**（两层解析、`range2re` 按位切割、切片、`[ab]` 字面量
   陷阱、空集语义、优先级），而非存储形态；真正防止「触发不同进程」的手段是**以 gsekit 语料
   为准的等价性回归测试**。
