## 需求背景

### 业务背景

在一次线上发布中，用户先通过灰度分组把版本 A（release 27264）发布给命中标签且落入 20% 灰度桶的实例，随后想用版本 B（release 27250）对同一批服务做全量发布。结果发现全量发布能覆盖大部分实例，却始终有少量实例（本例 2 个 pod）被"钉死"在旧版本 A，无法切到全量版本 B。

根因是版本匹配的选择规则：feed-server 为每个实例计算出它命中的所有分组，再从命中集合里**选择灰度比例（gray_percent）最大的分组**对应的版本。非灰度分组（全量意图）在该规则下灰度比例被视为 0，优先级最低，因此**永远输给任何残留的灰度分组**。只要旧灰度分组还挂在旧版本上、且实例又落入其灰度桶，全量发布就无法覆盖这些实例，形成"灰度锁死"。

同时，产品侧当前对灰度分组能力有两处约束，放大了该问题并限制了用户的正常操作：
- 创建/更新分组时灰度比例只允许 1–99，无法表达"灰度 100%（即全量）"；
- 一次发布中禁止同时绑定"含灰度比例的分组"与"不含灰度比例的分组"。

用户希望修复锁死，并从产品语义上把"非灰度分组"和"灰度 100%"统一为"对命中标签的全部实例下发"，让全量发布能够正确覆盖灰度残留。

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 非灰度分组在版本匹配选择中视为灰度 100%（最高优先级），使全量发布可覆盖残留灰度分组实例 | P0 | 发布用户 | 必须 |
| F-002 | 放开分组灰度比例创建/更新校验，允许灰度比例 1–100 | P1 | 发布用户 | 应该有 |
| F-003 | 放开发布绑定校验，允许同一次发布同时包含"含灰度比例"与"不含灰度比例"的分组 | P1 | 发布用户 | 应该有 |

### 详细功能描述

#### [F-001] 全量/非灰度分组覆盖灰度残留

- **输入**：一个实例在拉取/心跳时，同时命中"非灰度（全量）分组（指向新版本）"与"残留灰度分组（指向旧版本）"。
- **处理逻辑**：
  1. 实例命中多个分组时，非灰度分组在选择时按"灰度 100%"参与比较（等价于优先级最高）。
  2. 因此非灰度全量分组（100%）优先于残留灰度分组（如 20%），实例最终生效为全量分组指向的新版本。
- **输出**：实例切换到全量发布的目标版本，不再被旧灰度版本钉死。
- **边界条件**：
  - 实例只命中灰度分组、未命中任何非灰度分组时，维持原灰度选择逻辑不变。
  - 多个非灰度分组同时命中时，沿用现有"并列时按最近更新"的兜底规则。
- **异常处理**：
  - 实例不命中任何自定义/调试分组时，回落 default 兜底分组（行为不变）。

#### [F-002] 允许创建灰度比例 100 的分组

- **输入**：用户创建/更新分组时填写灰度比例值。
- **处理逻辑**：允许的取值范围由 1–99 放宽为 1–100；灰度比例 100 语义为"对命中标签的全部实例下发"。
- **输出**：灰度比例=100 的分组创建/更新成功。
- **边界条件**：小于 1 或大于 100 仍拒绝；灰度比例匹配符仍必须为"等于"。
- **异常处理**：非法取值返回明确的校验错误提示（提示范围更新为 1–100）。

#### [F-003] 允许灰度组与非灰度组混绑

- **输入**：用户在一次发布中同时选择含灰度比例与不含灰度比例的分组。
- **处理逻辑**：取消"若含灰度分组则所有分组必须含灰度比例"的阻断；非灰度分组按"灰度 100%"语义参与后续匹配。
- **输出**：混合分组的发布提交成功。
- **边界条件**：多个灰度分组之间"非灰度标签一致性"的既有校验是否保留，待确认（见未解决问题 Q-001）。
- **异常处理**：保留必要的合法性校验，非法组合给出明确错误提示。

## 非功能需求

### 性能需求

- 版本匹配逻辑改动不得增加单实例匹配的时间复杂度量级（维持与命中分组数线性相关）。

### 兼容性

- **行为兼容**：仅命中灰度分组的场景，选择结果保持不变；仅命中非灰度分组的场景，选择结果保持不变。行为变化仅发生在"同时命中非灰度分组与灰度分组"的交叉场景。
- **数据兼容**：不改变 released_groups / groups / strategies 表结构；存量数据无需迁移。

## 业务规则

### 业务逻辑规则

- **规则 R-001**：非灰度分组 = 对该分组标签命中的全部实例下发配置（标签范围内全量），不影响标签范围外的实例；不等同于"全部实例全量（All=true）"清空其它分组映射的发布。
- **规则 R-002**：非灰度分组在版本匹配选择中的优先级等价于灰度比例 100%。
- **规则 R-003**：灰度比例合法范围为 1–100（含 100）。

### 数据校验规则

- **灰度比例取值**：1 ≤ 灰度比例 ≤ 100；匹配符必须为"等于"。

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 某实例同时命中一个指向旧版本的灰度分组（如 20%）和一个指向新版本的非灰度全量分组 When 用户完成新版本的全量（非灰度）发布 Then 该实例最终生效版本为新版本（全量分组指向的版本），不被旧灰度版本钉死。
- [ ] **AC-002**：Given 某实例仅命中灰度分组 When 匹配版本 Then 选择结果与改动前一致（灰度逻辑不变）。
- [ ] **AC-003**：Given 某实例仅命中非灰度分组 When 匹配版本 Then 生效为该非灰度分组指向的版本（行为不变）。
- [ ] **AC-004**：Given 用户创建/更新分组并填写灰度比例=100 When 提交 Then 创建/更新成功。
- [ ] **AC-005**：Given 用户填写灰度比例=0 或 >100 When 提交 Then 拒绝并提示合法范围为 1–100。
- [ ] **AC-006**：Given 用户在一次发布中同时选择含灰度比例与不含灰度比例的分组 When 提交发布 Then 提交成功，不再报"所有分组必须含灰度比例"。

## 边界范围

### 本期包含

- F-001 非灰度分组视为灰度 100%、全量覆盖灰度残留（新发布/新匹配行为）。
- F-002 放开灰度比例创建/更新校验至 1–100。
