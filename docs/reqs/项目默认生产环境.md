## 需求背景

### 业务背景

BSCP 正在承接游戏业务场景，需要引入**项目（Project）**和**环境（Environment）**两个维度来管理配置。当前系统中 Project 和 Environment 的数据模型已经定义完成（`projects` 表和 `environments` 表均已存在），但创建项目时尚未自动关联环境，导致新项目创建后没有可用的环境维度。

核心痛点：
- 新项目创建后缺少默认环境，用户需要手动创建才能使用完整的环境维功能
- 游戏业务场景对"生产环境"有强依赖，每个项目天然需要一个生产环境作为基础
- 当前 `CreateProject` 接口仅处理项目本身，未联动创建关联的默认环境

### 用户故事

> 作为 BSCP 平台的使用者
> 我想要在创建项目时自动获得一个默认的生产环境
> 以便无需手动操作即可直接使用环境维度进行配置管理

## 功能需求

### 核心功能点

| 功能编号 | 功能描述 | 优先级 | 涉及角色 | 备注 |
|---------|---------|--------|---------|------|
| F-001 | 创建项目时自动创建默认生产环境 | P0 | 系统（后端内部逻辑） | 必须实现 |
| F-002 | 默认环境属性按约定值初始化 | P0 | 系统 | 名称=Default, 类型=prod, protected=true |
| F-003 | 项目与环境在同一事务中原子性创建 | P0 | 系统 | 任一失败整体回滚 |

### 详细功能描述

#### [F-001] 自动创建默认生产环境

- **输入**：用户调用 CreateProject API 创建项目
- **处理逻辑**：
  1. 系统接收 CreateProject 请求，执行原有的项目校验逻辑
  2. 项目数据写入 `projects` 表成功后，立即在同一事务内构建默认 Environment 记录
  3. 将默认 Environment 写入 `environments` 表，`project_id` 关联到刚创建的项目 ID
  4. 提交事务，返回项目创建结果
- **输出**：CreateProject 原有响应结构（不包含环境信息）
- **边界条件**：
  - 仅对新创建的项目生效，不处理历史存量项目
- **异常处理**：
  - 环境创建失败 → 整个事务回滚，项目也创建失败，返回错误信息

#### [F-002] 默认环境属性约定值

默认 Environment 的字段值如下：

| 字段 | 值 | 说明 |
|------|-----|------|
| name | `"Default"` | 固定名称 |
| type | `"prod"` | 生产环境类型 |
| memo | `""`（空） | 无额外描述 |
| protected | `true` | 标记为保护状态，不可删除 |
| display_order | `0` | 默认排序权重 |
| project_id | 关联的项目 ID | 外键关联 |
| tenant_id | 继承项目的 tenant_id | 与项目一致 |
| biz_id | 继承项目的 biz_id | 与项目一致 |

#### [F-003] 事务保证

- **输入**：CreateProject 调用链路
- **处理逻辑**：
  1. 开启数据库事务
  2. INSERT INTO projects ...
  3. INSERT INTO environments ... （使用步骤2返回的 project_id）
  4. COMMIT
  5. 如果步骤2或3任一失败 → ROLLBACK
- **输出**：事务提交成功或失败的整体结果
- **边界条件**：无
- **异常处理**：
  - 数据库连接异常 → 返回系统内部错误
  - 主键冲突 → 返回已存在的错误提示

### 可用性与稳定性

## 业务规则

### 业务逻辑规则

- **规则 R-001**：每个新创建的项目必须且只能自动创建一个默认环境（1:1 关系）
- **规则 R-002**：默认环境的 `type` 固定为 `"prod"`，不可通过此逻辑创建其他类型的环境
- **规则 R-003**：默认环境的 `protected` 标志为 `true`，表示不可被用户删除
- **规则 R-004**：如果项目创建失败，不应留下任何孤立的 environment 数据

### 数据校验规则

- **必填字段**：name, type, project_id, tenant_id, biz_id
- **格式要求**：
  - name：非空字符串
  - type：枚举值，必须为 prod / staging / test / dev 之一
- **取值范围**：type = "prod"（固定值）

### 权限规则

- 默认环境的权限继承自项目创建者，与项目权限保持一致

#### CreateProject 变更说明

```
变更前:
  Client → CreateProject → Insert(project) → 返回 {Project}

变更后:
  Client → CreateProject → BeginTx → Insert(project) → Insert(default_env) → Commit → 返回 {Project}
                                    ↓ (任意一步失败)
                                  Rollback → 返回错误
```

## 验收标准

### 功能验收

- [ ] **AC-001**：Given 用户调用 CreateProject API 且传入合法参数 When 项目创建成功 Then 系统自动在 environments 表中插入一条 name="Default"、type="prod"、protected=true 的环境记录，且该环境的 project_id 正确关联到新建项目
- [ ] **AC-002**：Given 用户调用 CreateProject API When 项目创建过程中环境写入失败 Then 整个操作回滚，projects 表中不存在该项目记录，environments 表中也不存在孤立的环境记录
- [ ] **AC-003**：Given 用户查询刚创建的项目及其关联环境 When 调用 ListEnvironments 接口传入该项目 ID Then 返回结果中包含一条 Default 生产环境
- [ ] **AC-004**：Given 用户尝试删除自动创建的默认环境 When 调用 DeleteEnvironment 接口 Then 返回错误提示（受保护的资源不允许删除）
- [ ] **AC-005**：Given 已存在的存量项目（需求上线前创建）When 查询这些项目的环境列表 Then 不自动补建环境，返回空列表或已有的人工创建的环境

### 性能验收

- [ ] **AC-P01**：CreateProject 接口的 P99 响应时间较改造前增长不超过 30ms

### 安全验收

- [ ] **AC-S01**：默认环境的访问权限与项目创建者的项目权限一致

## 边界范围

### 本期包含

- CreateProject 后端逻辑增加自动创建默认生产环境的功能
- 默认环境属性按固定值初始化（name=Default, type=prod, protected=true）
- 项目与环境在同一事务中原子性创建
- 对现有 CreateProject API 保持向后兼容（请求/响应结构不变）

### 本期不包含

- 不处理存量项目的环境补建（仅对新创建的项目生效）
- 不提供自定义初始环境名称/类型的配置能力
- 不修改前端界面或交互流程
- 不涉及环境的多语言支持（name 使用英文 "Default"）
- 不提供环境创建失败后的重试机制（直接事务回滚）
