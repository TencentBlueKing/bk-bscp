# Feature Specification: 集群环境类型、服务实例名等拓扑字段同步 CMDB 状态

**需求 ID**: 136320799（TAPD: 1020451610136320799）
**Feature Directory**: `specs/stories/136320799`
**创建时间**: 2026-07-21
**状态**: Draft
**来源**: `specs/stories/136320799/req.md`；澄清结论见 `specs/stories/136320799/questions.md`

## 概述

bscp「进程配置管理」模块提供「一键同步状态」能力，从 CMDB 拉取业务进程拓扑并落库。当前
缺陷：用户在 CMDB 修改了服务实例名称或集群环境类型后，在 bscp 点击「一键同步」，进程列表中
这些字段仍保持旧值，无法随 CMDB 更新，与对标产品 gsekit 存在能力差距。

本需求交付三件事：修复两个字段（服务实例名称 `service_name`、集群环境类型 `environment`）的
增量同步缺陷（F-001）；产出 gsekit 与 bscp 的 CMDB 同步能力差异对比清单（F-002）；补齐清单中
已确认的缺口（F-003）。

> **范围决策（2026-07-21，用户确认）**：集群名称（`set_name`）、模块名称（`module_name`）
> **不做 diff、不更新**，本期只同步集群环境类型与服务实例名称两个字段。故本 spec 中「拓扑字段」
> 特指这两个字段。

## User Scenarios & Testing

### 主要用户故事

作为进程配置管理的使用者，我在 CMDB 修改了集群环境类型、服务实例名称后，
希望通过 bscp「一键同步」让进程列表中的这些字段刷新为 CMDB 最新值，以保证 bscp 展示的进程信息
与 CMDB 保持一致，不再出现陈旧数据。

作为产品/研发负责人，我希望得到一份 gsekit 与 bscp 在 CMDB 同步能力上的差异对比清单，以明确
bscp 还有哪些同步项不支持，规划后续补齐。

### 验收场景（Acceptance Scenarios）

1. **AC-001（集群环境类型更新）**：
   **Given** CMDB 中某进程所属集群的环境类型由「测试」改为「正式」，
   **When** 用户在 bscp 进程配置管理页点击「一键同步状态」并等待同步完成，
   **Then** 该进程在 bscp 进程列表中的集群环境类型展示为「正式」，与 CMDB 当前值一致。

2. **AC-002（服务实例名称更新）**：
   **Given** CMDB 中某进程所属服务实例名称被修改，
   **When** 用户点击「一键同步状态」并等待同步完成，
   **Then** 该进程在 bscp 进程列表中的服务实例名称展示为修改后的名称。

3. **AC-003（集群名称、模块名称不更新）**：
   **Given** CMDB 中某进程所属集群名称、模块名称被修改（环境类型与服务实例名称不变），
   **When** 用户点击「一键同步状态」并等待同步完成，
   **Then** 该进程不因集群名称/模块名称变化产生更新（本期范围外），行为与现状一致。

4. **AC-004（无变化不产生无意义更新）**：
   **Given** 集群环境类型与服务实例名称两个字段均未变化，
   **When** 用户点击「一键同步状态」，
   **Then** 不产生无意义的更新，其余同步行为（实例扩缩容、进程状态、别名/进程属性等）与现状一致。

5. **AC-005（对比清单交付）**：
   **Given** 已完成 gsekit 与 bscp 两侧 CMDB 同步能力梳理，
   **When** 查看交付物，
   **Then** 存在一份 gsekit vs bscp CMDB 同步能力对比清单，逐项标注 bscp 是否支持及差异点。

6. **AC-006（缺口补齐生效）**：
   **Given** 对比清单中标注为「需补齐」的缺口，
   **When** 完成本期开发，
   **Then** 这些缺口在 bscp 一键同步中生效，并可通过对应用例验证。

### 边界与异常情况（Edge Cases）

- **字段被清空**：当 CMDB 侧某字段被清空（如服务实例名称为空）时，bscp 直接以 CMDB 值覆盖
  （含覆盖为空），对齐 gsekit 对标基准，不做「空值保留旧值」的保护（澄清结论 Q-002）。
- **别名与拓扑字段同时变更**：进程别名变更且触发进程记录复用/恢复的场景，集群环境类型与
  服务实例名称两个字段也应刷新为 CMDB 最新值，不残留旧值。
- **单条记录同步失败**：沿用现有一键同步的错误处理与任务结果反馈机制，本需求不新增机制。
- **多条链路一致性**：一键同步、定时全量同步、CMDB watch 增量同步共用同一套进程 diff 逻辑；
  本需求以「一键同步」为验收入口，diff 逻辑修复后其余链路一并受益。

## Requirements

### 功能需求（Functional Requirements）

- **FR-001**：一键同步时，系统 MUST 对 CMDB 中已存在（进程主键匹配）的进程记录，识别并更新
  以下两个字段的变化：服务实例名称（`service_name`←CMDB 服务实例 `name`）、集群环境
  类型（`environment`←CMDB `bk_set_env`）。（对应 req.md F-001）

- **FR-001a**：集群名称（`set_name`）、模块名称（`module_name`）**不在本期同步范围**，系统
  MUST NOT 因这两个字段的变化产生进程更新，保持现状（不做 diff、不写回）。（范围决策 2026-07-21）

- **FR-002**：两个字段的更新 MUST 采用「直接以 CMDB 值覆盖」语义，包括 CMDB 侧值为空时
  覆盖为空，不保留旧值。（澄清结论 Q-002）

- **FR-003**：集群环境类型取值 MUST 与 CMDB `bk_set_env` 保持一致，取值范围为 `1`（测试）/
  `2`（体验）/ `3`（正式）。（对应 req.md 数据校验规则）

- **FR-004**：当集群环境类型与服务实例名称两个字段均未变化时，系统 MUST NOT 因这两个字段产生进程
  更新；同步的其余既有行为（实例扩缩容、进程状态、别名/进程属性、os_type、agent 状态等）MUST 保持
  与现状一致，不被本需求改变。（对应 req.md F-001 边界 + AC-004）

- **FR-005**：当进程别名变更并触发进程记录复用/恢复分支时，系统 MUST 将集群环境类型与服务实例
  名称两个字段一并刷新为 CMDB 最新值，不残留旧值。（对应技术澄清 TR-001）

- **FR-006**：系统 MUST 产出一份 gsekit 与 bscp 在 CMDB 同步能力上的差异对比清单（Markdown），
  逐项对比两侧「同步哪些实体/字段」「新增与增量更新分别覆盖哪些字段」，并标注 bscp 是否支持及
  与 gsekit 的差异点。交付物位于 `docs/` 下（具体路径实现时确定）。（对应 req.md F-002）

- **FR-007**：系统 MUST 对 F-006 对比清单中标注为「bscp 缺失且需补齐」的缺口进行本期补齐，
  使其在 bscp 一键同步中生效。补齐范围以对比清单结论 + 用户确认为准，不超出 gsekit 一键同步的
  能力边界。（对应 req.md F-003；范围确认见 Q-003）

- **FR-008**：本需求 MUST NOT 新增 CMDB 接口调用或新的同步触发入口，MUST 复用 bscp 现有 CMDB
  同步链路与客户端。（对应 req.md 约束条件）

### 关键实体（Key Entities）

- **进程主记录（bscp `processes` / `ProcessSpec`）**：本需求关注的字段——
  - `service_name`：服务实例名称，来源 CMDB 服务实例 `name`（本期同步）
  - `environment`：集群环境类型（"1"/"2"/"3"），来源 CMDB `bk_set_env`（本期同步）
  - `set_name`：集群名称，来源 CMDB `bk_set_name`（**本期不同步**）
  - `module_name`：模块名称，来源 CMDB `bk_module_name`（**本期不同步**）
  上述字段均已存在于数据模型中，本需求不涉及数据模型（表结构）变更。

- **CMDB 进程相关信息（`process_related_info`）**：进程所属集群、模块、服务实例及其名称/环境
  类型的权威数据源，bscp 已有客户端，本需求不新增接口。

## Success Criteria

- **SC-001**：在 CMDB 修改集群环境类型或服务实例名称并执行一次「一键同步」后，bscp 进程列表中
  对应字段展示值与 CMDB 当前值一致（对应 AC-001~002）。
- **SC-002**：集群环境类型与服务实例名称无变化时，「一键同步」不会因这两个字段产生更新，其余同步
  行为无回归；集群名称/模块名称变化不触发更新（对应 AC-003~004）。
- **SC-003**：交付一份 gsekit vs bscp CMDB 同步能力对比清单，逐项可查证支持情况与差异（对应
  AC-005）。
- **SC-004**：对比清单中确认需补齐的缺口在本期完成后于 bscp 一键同步中生效，并有对应用例验证
  （对应 AC-006）。

## Assumptions（假设与默认）

- 集群环境类型与服务实例名称的空值处理采用「直接覆盖」策略（Q-002 已澄清）。若产品后续要求空值
  保护，将覆盖此结论并重新评估。
- 集群名称/模块名称不同步是明确的范围决策（2026-07-21 用户确认）；虽 gsekit 会随 expression 刷新
  这两项，bscp 本期有意不对齐，如后续需要再作为独立需求评估。
- F-003 缺口补齐的具体范围依赖 F-002 对比清单产出后确认（Q-003 非阻塞）；F-001 根因明确、可独立
  实现与验收，先行推进。若对比清单发现较多额外缺口，需回到需求侧补充范围并重估工时。
- 需求优先级（Q-001）为产品排期决策，非技术阻塞，不影响本 spec 收敛。
- 对标基准以 gsekit 一键同步（`sync_biz_process`）字段同步行为为准（对标限制：补齐范围不超出
  gsekit 一键同步能力边界）。

## 范围（Scope）

### 本期包含

- 一键同步时服务实例名称、集群环境类型两个字段的增量更新（F-001）
- gsekit 与 bscp CMDB 同步能力对比清单（F-002）
- 对比清单中已确认缺口的补齐（F-003，范围以对比结论为准）

### 本期不包含

- **集群名称（`set_name`）、模块名称（`module_name`）的同步更新**（范围决策 2026-07-21）
- 进程状态、进程实例扩缩容、别名、进程属性等既有同步逻辑的改造
- 新增 CMDB 接口调用或新的同步触发入口
- gsekit 侧任何改动

## 依赖（Dependencies）

- 蓝鲸 CMDB（HTTP/APIGW）进程相关信息聚合查询（`process_related_info`）：bscp 已有客户端，本需求
  不新增接口。集群环境类型与服务实例名称均已在返回体内。
